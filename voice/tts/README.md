# lab-assistant TTS (gepard-1.0)

The text-to-speech stage. A FastAPI service (`tts/server.py`) that wraps
**gepard-1.0** (a codec-based LM TTS) plus the **NVIDIA NeMo NanoCodec**
vocoder, running on the GPU host. It powers both the assistant's spoken replies
and the in-app **TTS playground**.

- Output: **22050 Hz mono WAV (PCM16)**.
- Languages: **English is strongest**; also Spanish (Mexico), Portuguese (Brazil),
  Dutch. No Chinese.
- Runs the reference PyTorch path, not gepard-vllm (which needs CUDA 13 / Blackwell).

---

## Using the playground (browser)

1. Open the app (your tunnel URL, e.g. `https://<your-tunnel-host>`) and click **🔊 TTS** in the header.
2. Type any text, pick a **Voice**, optionally tune **Temp / CFG / Top-k**, and hit
   **Synthesize**. The audio plays inline. It is independent of the mic conversation
   (no ASR, no LLM), so you can A/B voices and settings freely.

### Controls

| Control | Range (default) | What it does |
|---|---|---|
| **Voice** | one of 19 (`default`) | Speaker identity. `default` is the model's own voice; the rest are preset speakers (below). |
| **Temp** (temperature) | 0.05-1.5 (**0.3**) | Sampling randomness. Lower = steadier, more consistent delivery. Higher = more expressive but more artifacts. 0.2-0.4 is the sweet spot. |
| **CFG** (cfg_scale) | 1.0-3.0 (**1.0**) | Text guidance. **Keep at 1.0 for normal text.** Values >1 only help *very short* phrases; on longer text they rush delivery and flatten prosody. |
| **Top-k** | 0-100 (**0**) | 0 = disabled (sample full distribution). >0 keeps only the top-k tokens (more conservative). |

## Voices

`default` plus 18 presets, provisioned into `tts/voices/` by `deploy/setup-tts.sh`,
which runs `tts/fetch_voices.py` when that directory arrives with no `.pt` files. The
download pulls `speakers/*.pt` from the public HuggingFace Space `nineninesix/gepard`;
no `HF_TOKEN` is needed since the Space is public. `deploy/sync-to-host.sh` rsyncs any
`.pt` files already present on the dev machine, so a host that received them that way
fetches nothing. Names are the `.pt` filenames; descriptions are approximate.

**English (US):** `en_andrew`, `en_oak` (Oakland), `en_boston`, `puck`, `expresso_f`
(expressive female), `den`, `elon` (novelty)
**English (UK):** `scottish`, `scottish_1` (Scottish), `scouse_enh` (Liverpool);
`italian` (Italian-American accent)
**Spanish (Mexico):** `mex_m`, `mx_f`
**Portuguese (Brazil):** `br_m`, `br_f`
**Dutch:** `nl_m`, `nl_f`, plus `nurisa`

`default` is provisioned differently from the 18 presets above: it is pinned to
`tts/voices/default.pt`, generated by `tts/make_gepard_voice.py` rather than
downloaded, and it is the one voice file committed to the repo. No deploy script
regenerates it, because regenerating re-rolls the pinned speaker (`--method capture`
freezes one unconditioned generation's tokens at temperature 0.3, which is
overwrite-safe but not content-stable across runs); `tts/fetch_voices.py` skips it by
name for the same reason. Voices are read from `tts/voices/` once at process startup,
so a new `.pt` file dropped in or fetched needs a TTS restart before it shows up in
`/voices`.

### Adding a voice

To add one of the 18 presets by hand (normally unnecessary; `deploy/setup-tts.sh` already
does this), drop a reference-code file into `tts/voices/` on the GPU host and restart TTS;
it appears in `/voices` by filename. Format (per the gepard SpeakerLibrary), saved with
`torch.save`:

```python
{ "ref_codes": <LongTensor [1, T_ref, C_total]>,  # unfolded NanoCodec codes
  "name": "...", "sample_rate": 22050, "source": "clip.wav" }   # metadata optional
```

A bare `[1, T, C]` LongTensor is also accepted. (Cloning a *new* voice from a short
audio clip = encode the clip with NanoCodec then unfold to these codes; not yet wired
into the playground.)

## HTTP API (localhost on the GPU host, port 8040)

The TTS service is not publicly exposed: it binds to `127.0.0.1` and the browser
reaches it through the orchestrator. For direct testing on the GPU host:

```bash
# list voices
curl -s http://localhost:8040/voices

# synthesize
curl -s -X POST http://localhost:8040/synthesize \
  -H 'Content-Type: application/json' \
  -d '{"text":"Hello there.","voice":"en_andrew","temperature":0.3,"cfg_scale":1.0,"top_k":0}' \
  -o out.wav
```

| Endpoint | Request | Response |
|---|---|---|
| `GET /health` | - | `{"status":"ok"\|"loading","device","model","voices":<n>}` |
| `GET /voices` | - | `{"voices":["default", ...]}` |
| `POST /synthesize` | `{text, voice?, temperature?, cfg_scale?, top_k?, max_frames?}` | `audio/wav` (22050 Hz PCM16); `400` empty text, `503` loading, `500` synth error |

`max_frames` (default 1075 ≈ 50 s at 21.5 fps) caps output length; rarely changed.

## WebSocket messages (through the web app / tunnel)

The playground uses the same socket as the chat:

- Client -> `{"type":"list_voices"}` -> Server `{"type":"voices","voices":[...]}`
- Client -> `{"type":"tts_test","text":...,"voice":...,"temperature":...,"cfg_scale":...,"top_k":...}`
  -> Server `{"type":"tts_test_audio","audio_b64":...,"sample_rate":22050,"format":"wav"}`
  or `{"type":"tts_test_error","text":...}`

## Operations

- **Config (env):** `LA_TTS_HOST` `LA_TTS_PORT` (8040) `LA_TTS_DEVICE` (cuda)
  `LA_TTS_MODEL` `LA_TTS_CODEC` `LA_TTS_VOICES_DIR` `LA_TTS_TEMP` `LA_TTS_CFG`
  `LA_TTS_TOPK` `LA_TTS_MAXFRAMES`.
- **Restart:** the services run as systemd `--user` units installed by
  `deploy/install-services.sh`, so restart with
  `systemctl --user restart lab-assistant-tts`, and read the log with
  `journalctl --user -u lab-assistant-tts -n 50 --no-pager -o cat`. Model + voices load
  at startup (~1 min). For a throwaway detached run instead, `bash deploy/run-services.sh`
  relaunches both GPU services (logging to `logs/tts.log`) and `bash deploy/stop-services.sh`
  stops them. `bash deploy/health.sh` probes both.
- **Model cache:** `$LA_HF_HOME` (default `~/lab-assistant-hf`), exported as `HF_HOME` by
  `deploy/env.sh` and by the systemd unit, so every service shares one cache.
- **Voices on a fresh host:** `deploy/sync-to-host.sh` rsyncs `tts/voices/*.pt` into
  `$LA_APP_DIR`, and `deploy/setup-tts.sh` fetches the 18 presets only when the directory
  arrives with no `.pt` files. It never generates `default.pt`, which ships committed with
  the repo, so a deploy cannot re-roll the pinned reference voice. Setup warns if
  `en_oak.pt` is absent: a missing voice file makes gepard fall back to an unconditioned
  speaker that drifts between sentences, and it reports no error when it does.
- **Latency:** ~2 s to synthesize a sentence on the L4 (reference PyTorch path).
- **Make the chat use a preset voice** (not just the playground): set `LA_TTS_VOICE`
  or pass a `voice` in the orchestrator's chat synth call.
